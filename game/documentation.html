Player Class

The Player class takes in the following:
  • colour //Simply the players colour
  • name //The player name for visual purposes (scoreboards, etc)
  • id //The id of the player for backend in the code, typically auto-assigned, so dont worry too much outside of testing
  • isBot //Simple boolean if player is human or bot

Player methods consist of the following:
  • getOwned() //Returns a list of the planetos owned by the specific player
  • getColour() //Irrelevant for most cases, unless you make a racist bot
  • getID() //Returns the players ID number
  • getScore() //Returns the players score

Planeto Class

The Planeto class has the following attributes (relevant ones listed with method to access):
  • ownerID - Accessed using getOwner() //Owner ID of the player
  • x - Accessed using getX() //X coordinate of planeto
  • y - Accessed using getY() //Y coordinate of planeto
  • connections - Accessed using getConnections() //The list of connections, as integers
  • hasShield - Accessed using getShield() //Boolean tracking the shield on the planeto
  • value - Accessed using getValue() //Integer holding point value of planeto
  • bonerShowing - Accessed using getShowing() //The only way to tell if a planeto is part of a formation, boolean (explained below)
  • lockLife - Accessed using getLockLife() //Number of turns the planeto is locked for, used to tell if planeto is locked
  • prevOwner - Accessed using getPrevOwn() //Previous owner ID of the planeto

Further explanation of bonerShowing:

bonerShowing is used for the sake of the bots being able to see formations, but only when they're showing, otherwise the formation is invisible code-wise. Of course, methods/variables can be made for the bot to remember where they are though...
